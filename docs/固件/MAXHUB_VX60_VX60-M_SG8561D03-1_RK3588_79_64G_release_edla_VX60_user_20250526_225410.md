## 🔍 二、核心区别（Key Differences）

### 1. **镜像组织方式不同**

| 类型 | OTA 固件 | USB 升级固件 |
|------|--------|-------------|
| **分区模型** | **A/B 分区或动态分区（Dynamic Partitions）** | **传统静态分区 + Super 分区打包** |
| **关键镜像** | 多个独立分区镜像：`system.img`, `vendor.img`, `product.img`, `system_ext.img` 等 | **单一 `super.img`（2.4GB）** 包含所有逻辑分区 |
| **技术背景** | Android 10+ 推荐使用 **动态分区（Dynamic Partitions）**，OTA 时可单独更新某个分区 | USB 升级采用 **整块烧录模式**，将整个 `super` 分区一次性写入 eMMC |

> 📌 注：`super.img` 是 Android 动态分区的容器镜像，内部通过 LPT (Logical Partition Table) 划分为 `system`, `vendor`, `product` 等子分区。

---

### 2. **升级机制不同**

| 维度 | OTA 升级 | USB 升级 |
|------|--------|--------|
| **触发方式** | 通过 Web 页面上传 `.zip` 包（内含这些 `.img` 文件） | 将固件目录放到 U 盘根目录，设备开机自动检测 |
| **更新粒度** | **分区级更新**：可只更新 `system.img` 或 `vendor.img`（支持差分/增量） | **全量覆盖**：必须重写整个 `super.img`（即使只改了一个 app） |
| **依赖系统状态** | 需要当前系统能正常启动并运行 Web 服务 | 可在 Recovery 或 Bootloader 模式下执行，**不依赖当前系统** |
| **适用场景** | 日常维护、小版本迭代 | 首次刷机、系统崩溃恢复、产线烧录 |

## 🔧 三、技术原理深入解释

### ▶️ 为什么 OTA 不用 `super.img`？
- OTA 升级通常由 **update_engine** 或 **Recovery** 执行，它需要**解包 `super.img` 并只更新变更的子分区**。
- 若直接传 `super.img`，则无法做**块级差分（block-based delta）**，会浪费带宽。
- 因此 OTA 包保留原始分区镜像，便于生成 **增量包（incremental OTA）**。

### ▶️ 为什么 USB 升级用 `super.img`？
- USB 升级本质是 **“烧录”而非“更新”**，工具（如 `rkdeveloptool` 或 `upgrade_tool`）直接将 `super.img` 写入 eMMC 的 `super` 分区。
- 这种方式**简单、可靠、兼容性好**，适合出厂或救砖场景。
- 同时保留 `boot`, `uboot`, `logo` 等独立分区，因为它们不在 `super` 范围内。

## ✅ 四、总结：如何选择？

| 场景 | 推荐方式 |
|------|--------|
| **首次刷机 / 设备变砖** | ✅ USB 升级（完整、可靠、不依赖系统） |
| **日常版本升级（如 V1.0 → V1.1）** | ✅ OTA 升级（节省流量、可增量、用户友好） |
| **产线批量烧录** | ✅ USB 升级（配合脚本自动化） |
| **调试分区内容** | ✅ OTA 镜像（可单独提取 `system.img` 分析） |

> ⚠️ 注意：**不能混用**！
> - 不要把 OTA 的 `system.img` 直接用于 USB 升级工具（会失败）；
> - 也不应把 `super.img` 上传到 Web OTA 接口（系统无法解析）。
